日志采集方案II

背景

早期采用EFK（Eleasticsearch+Flume+kibana）的架构模式，在各服务器端部署flume代理，以tailf的方式，采集日志文件，当日志文件量大时，会导致cpu的使用率会上升，主要是flume代理会占用服务器端资源，基于上述原因，需求更好的解决方式。查阅官网发现，filebeat,一个轻量级的采集程序，基于golang开发，占用资源比flume的少很多。下面主要介绍filebeat+logstash+Eleasticsearch的日志采集方案。

简介

五大组件

  • filebeat：部署在采集端，负载采集，轻量，占用资源相对较少
  • logstash：部署在服务端，主要汇集和过滤各个采集端的数据
  • Flume：可做为采集端和采集服务端，在大数据量时，会占用宿主资源
  • Eleasticsearch：作为数据的存储，具备高可用，高扩展性的服务
  • kibana：数据展现层，提供日志的可视化分析

架构

  1. flume-agent(采集端)=》flume-server（服务端）=》Eleasticsearch(集群)+kibana

    优点：
    具有分布式、可靠、可用的特性，可水平扩展。

    缺点：
    数据量大时，比较耗资源

  2. filebeat（采集端）=》logstash（服务端）=》Eleasticsearch(集群)=》kibana

    优点：
    轻量，体积小，占用资源少

  3. filebeat（采集端）=》消息队列
    消息队列=》logstash（服务端）+Eleasticsearch(集群)+kibana
    优点：
    引入消息队列，filebeat 支持 Kafka、Redis消息队列；
    在数据量大和高并发的情况下，均衡了网络传输，降低了网络闭塞。

日志规范约定

原创
2017-07-06
java
java

1.目标

日志的要求，希望能够按日期、文件大小切割，并能够自动压缩。并自动删除约定天数后

的备份日志。

2.简介

查阅官方资料，目前log4j 1.x没办法达到以上目标，如需要需自行进行扩展。

还好log4j2 和logback都提供了这些功能，在性能上log4j2有了大大的提升，采用异步写，比logback快18倍。

log4j2 还提供了对log4j的适配，并且无需更改代码即可轻松更新至log4j2。基于以上，提供log4j2和logback两种

通用日志配置,使其达到目标效果。

3.log4j升级log4j2(无需改动代码)

官网原文：https://logging.apache.org/log4j/2.x/index.html

log4j升级log4j2无需更改代码，只需修改maven相关的依赖,并添加log4j2.xml至classpath下。Maven配置如下：

官网升级说明： https://logging.apache.org/log4j/2.x/maven-artifacts.html

2017-07-02
linux
linux

1.查看java线程

ps -ef | grep mrf-center | grep -v grep

2.查看耗cpu时间最长的进程

top -Hp pid

3.得到最长的线程ID的十六进制值

TIME列就是各个Java线程耗费的CPU时间，CPU时间最长的是线程ID为21742的线程，用

printf “%x\n” 21742 得到21742的十六进制值为54ee

4.打印堆线程

jstack pid |grep 54ee
打印线程堆栈信息 jstack pid |grep tid -A 30

jstat -gc 12538 5000 jstat -gcutil 3319 2000

centos7 防火墙操作命令备忘

原创
2017-07-01
linux
linux

1、firewalld的基本使用

启动： systemctl start firewalld

查看状态： systemctl status firewalld

停止： systemctl disable firewalld

禁用： systemctl stop firewalld

启动一个服务：systemctl start firewalld.service

关闭一个服务：systemctl stop firewalld.service

重启一个服务：systemctl restart firewalld.service

显示一个服务的状态：systemctl status firewalld.service

在开机时启用一个服务：systemctl enable firewalld.service

在开机时禁用一个服务：systemctl disable firewalld.service

查看服务是否开机启动：systemctl is-enabled firewalld.service

查看已启动的服务列表：systemctl list-unit-files|grep enabled

查看启动失败的服务列表：systemctl –failed

2.开启端口

添加

firewall-cmd –zone=public –add-port=80/tcp –permanent
（–permanent永久生效，没有此参数重启后失效）

重新载入

firewall-cmd –reload

查看

firewall-cmd –zone= public –query-port=80/tcp

删除
firewall-cmd –zone= public –remove-port=80/tcp –permanent

3.配置firewalld-cmd

查看版本： firewall-cmd –version

查看帮助： firewall-cmd –help

显示状态： firewall-cmd –state

查看所有打开的端口： firewall-cmd –zone=public –list-ports

更新防火墙规则： firewall-cmd –reload

查看区域信息: firewall-cmd –get-active-zones

查看指定接口所属区域： firewall-cmd –get-zone-of-interface=eth0

拒绝所有包：firewall-cmd –panic-on

取消拒绝状态： firewall-cmd –panic-off

查看是否拒绝： firewall-cmd –query-panic
